#### [代码就是工程，类型是科学](https://www.tweag.io/posts/2020-03-05-peirce.html) 

昨天提到的文章。编程到底是一项怎样的活动，真的可以从中找到乐趣吗。有时候，人们必须像工程师一样思考——在多重约束条件下找到问题的最佳解决方案。但是，有时候也必须像科学家一样思考——根据你所拥有的数据，建立实现目标的通用规则。这两种思维方式有着截然不同的性质，是不同的，但也是互补的。

这篇文章借用[Charles Sanders Peirce](https://en.wikipedia.org/wiki/Charles_Sanders_Peirce) 的推理理论来解释这两种模式。 Peirce将推理理论分为三个互补的过程：演绎、溯因和归纳 (deduction, *abduction* and *induction*)。一起来看下这三个过程分别是如何与软件工程相联系的。

- 演绎的结果就是事实的确定性: 如果我们得到 a，我们就一定得到 b。演绎的实际效用在于它能够缩小问题的范围: 如果我们想要实现 b，并且我们知道 a 意味着 b，我们可以将我们的目标改为 a 。
- 溯因的机制是这样的：如果我们知道一个事实 b，并且我们知道 a 意味着  b，那么我们就可以得出一个看似合理的结论。 比如a：如果错误得到了解决，就不该看到某些行为，b：我再也看不到些行为了，得出结论我解决了这个问题。
- 归纳的机制是这样的: 如果我们知道一个事实 b，并且我们知道一个事实 a，那么我们就可以把 a 导致 b 作为一个看似合理的结论。 

文章得出了以下几个结论：

- 我们做出的最明智的决定可能不是事实的逻辑结论——它们通常只是貌似合理的理论。 至少，它们不能是错误的，但是如果可能的话，我们必须利用手头的信息做出最好的决定。
- 溯因法和归纳法虽然简单，但却无处不在，识别出具体使用的方式，可以更容易知道如何找到弱点
- 确定在哪些地方使用了合理推理，可以显示哪些地方有可能发生错误

#### [一致性哈希——一个高效、可扩展的数据分配算法](https://medium.com/@animeshgaitonde/consistent-hashing-an-efficient-scalable-data-distribution-algorithm-a81fc5c0a6c7) 



一致性哈希算法用于解决数据分区问题，目前被 cassandra , Rika, Dynamo DB 等广泛使用。

能够使用达到 `O(1)` 读写的数据结构是 `HashMap` ,但是在分区存储的结构中，普通的 `HashMap` 结构存在一下两个问题：

- `hash(data)%N` 与 `N` 强相关，如果进行横向扩展，需要对之前的所有数据进行重新分配

- 依然与 `N` 相关，如果机器损坏需要被移除，依然要进行数据的重新分配

一致性哈希算法可以解决这类问题，算法绑定的是 server 的 ip 地址，对每一个 server ， 计算 `hash(ip)%360.00` ,同样对于数据，计算 `hash(data)%360.00`  ，即将hash值映射到圆上的一个点，之后按时钟顺序匹配数据最相近的 server ，这样带来的好处是显而易见的，增加或者减少 server 时，只有两个 server 之间的数据或者被减少的 server 上的数据需要重新分配。

同时为了进一步平衡每个 server 的负载，增加虚拟节点的概念，每个 server 都会映射几个虚拟节点，均匀分布在环上。
